# Failure analysis — baseline-val-v0

A_mean 48.8% · U90 21.2% · 125/252 records failing

Most failures come from the SQL writer omitting/misapplying NL constraints in aggregations and the SchemaLinker exposing incomplete column metadata; the Validator also aborts on common phrasing ambiguities instead of applying sensible defaults; the guardrail needs a friendlier, DB-aware refusal. Below are concrete, minimal guidance changes per responsible agent to reduce each failure class.

## wrong_result → SQLWriter
**Hypothesis**: The SQLWriter frequently omits or misapplies explicit natural-language filters and uses mismatched denominators when producing aggregates/percentages. Examples: the race completion query sent to Validator lacked the year and nationality filters asked by the user (validator fragment shows the SQL with no WHERE filtering for year or nationality), and percentage queries use COUNT(bond_id) or COUNT(*) as a denominator without ensuring the denominator matches the numerator's filtered population (exemplar percent SQL: "SUM(CASE WHEN bond_type = '=' THEN 1.0 ELSE 0.0 END) / COUNT(bond_id)"). These omissions yield correct SQL that aggregates over the wrong population.

**Fix direction**: Add an output-checklist to SQLWriter: (1) enumerate all explicit constraints from the question (dates, nationality, molecule id, funding type, 'only S' patterns, etc.); ensure each appears as a WHERE or JOIN predicate in the final SQL (e.g. WHERE races.year BETWEEN 2007 AND 2009 AND drivers.nationality = 'Japanese'). (2) For any percentage/ratio, build numerator and denominator from the same filtered set (preferred pattern: numerator = SUM(CASE WHEN <target_cond> THEN 1.0 ELSE 0.0 END), denominator = SUM(CASE WHEN <base_filter> THEN 1.0 ELSE 0.0 END)) and CAST to REAL to avoid integer division. (3) Before emitting SQL, render a 1-line provenance statement listing which NL constraints were translated to WHERE/JOIN clauses and require them to appear in the SQL; if a constraint is missing, add it rather than guessing a different population.

## wrong_tables → SchemaLinker
**Hypothesis**: SchemaLinker sometimes returns incomplete or inconsistent column metadata (describe_table vs sample_rows) and does not expose inferred column names/synonyms, causing the SQLWriter to reference wrong/missing columns or choose suboptimal tables. Example: describe_table for cards lists columns like asciiName but not name, yet SQLWriter produced "SELECT DISTINCT text FROM cards WHERE name = 'Benalish Knight';" — a mismatch between reported schema and used column names. Similar omissions/abbreviations appear in other exemplars where sample_rows headers include columns not shown in describe_table.

**Fix direction**: Change SchemaLinker guidance to a) always return both describe_table and the sample_rows header row, and b) when a column appears in sample_rows header but not in describe_table, add it to the schema metadata as an 'inferred' column (include original header spelling/casing). Also add a small synonym map for common variants (name, DisplayName, asciiName -> name) and expose it to SQLWriter. Finally, if describe_table and sample_rows disagree, tag the table as 'ambiguous' so the SQLWriter will prefer sample_rows names or ask a short clarification rather than guessing.

## no_sql_executed → Validator
**Hypothesis**: The Validator (and its pipeline usage) aborts or pushes for clarification on common, resolvable phrasing ambiguities instead of applying a reasonable default, which leaves many interactions with no SQL executed. Example: for '20% higher than average' the system asked whether the user meant AVG * 1.2 or some other formula (exemplar asked choice between two interpretations) and stopped rather than adopting a default and proceeding with a note. Other runs show the pipeline returning to SQLWriter without producing SQL.

**Fix direction**: Relax Validator behavior for a small set of frequent numeric and comparison ambiguities: adopt default mappings and proceed, while recording the assumption in the result. Concrete defaults to implement: 'X% higher than average' => threshold = AVG * (1 + X/100); 'difference' => absolute difference unless 'signed' is specified; 'from YEAR to YEAR' => inclusive BETWEEN. If Validator would have asked a clarification from the user, instead let it execute SQL using the default and add a one-line footnote about the assumed interpretation. Also ensure Validator returns clear execution errors (so SQLWriter can retry) rather than bouncing back to the orchestrator with no SQL attempt.

## guardrail_missed → guardrail
**Hypothesis**: The guardrail replies blunt refusals when user intent is outside the DB scope but does not first check whether the user's request could be reframed as a DB query or whether the DB contains relevant fields; this produces poor UX and unnecessary 'none' exec_match outcomes. Example: user asked for restaurant recommendations and the system immediately returned a hard refusal message without checking if the attached database had location/restaurant-like tables (final answer: "Sorry—I can’t help with restaurant recommendations...").

**Fix direction**: Modify guardrail policy to be DB-aware: before refusing, check SchemaLinker output for any tables/columns that could satisfy the request (location, name, address, category). If none exist, emit a short, helpful refusal that also suggests how to make the request answerable (e.g., 'I can only answer read-only analytics about the attached DB; if you provide a table or columns or ask a DB-related question I can help'). Only perform full refusal if the request is disallowed; otherwise prefer a helpful reframe.
